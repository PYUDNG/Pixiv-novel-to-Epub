## 先读 AGENT.md，再看代码

在理解或修改项目中**任何模块**的代码之前，必须优先检查该模块目录下是否存在 `AGENT.md`（`src/modules/<模块名>/AGENT.md`）。如果存在，**先完整阅读 AGENT.md**，了解模块的职责、架构、外部 API 和编码约定，然后才进入具体代码文件。

这确保你对模块的整体设计有正确理解，避免因遗漏上下文而做出错误判断。

---

## 每个模块都应创建 AGENT.md

每个独立模块（`src/modules/*/`）都应在自身目录下创建 `AGENT.md`，遵循以下结构：

```markdown
# 模块名 — 一句话职责

## 概述
模块的定位、解决什么问题、依赖哪些其他模块。

## 入口与生命周期
模块在 loader 中如何被激活：checkers 规则、enter/leave/toggle 钩子行为、loader外常态执行的代码有何功能逻辑。

## 文件结构
| 文件/目录 | 职责 |
|---|---|

## 外部 API（必需）
模块暴露给其他模块的接口：可调用的函数、可导入的 composable、可监听到的事件签名。**此段不可省略**，因为其他 agent 依赖它进行跨模块调用。可以完整写签名，也可指向含有完整签名的具体文件并附简单说明。

## 重要实现细节
架构决策、特殊处理、容易出错的坑。

## 编码约定
本模块内应遵守的规则（如是否使用 i18n、样式策略、icon 集选择等）。
这里仅仅写本模块特殊的规则，如果是整个项目都应该遵守的规则，则应该写成rule
```

**关键要求：**
- 「外部 API」段落不可省略——这是其他 AI agent 跨模块调用时的唯一可靠信息来源
- 类型签名应写完整（参数名、类型、返回值），避免 agent 猜测
- 如果代码内类型签名和JSDoc本身就很完整，可以不用重复写，而是指向含有签名的具体文件，并附简单说明
- 如果模块当前没有外部接口，也需写明「当前无外部调用接口」
- 各段落详略可以按模块复杂度调整，但以上 6 个段落的标题建议保留
